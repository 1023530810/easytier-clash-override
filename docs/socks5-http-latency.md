# 为什么通过 SOCKS5 访问内网 HTTP 服务会卡？

## 问题现象

通过 Clash 的 SOCKS5 代理访问 EasyTier 虚拟网络中的 HTTP 服务（如 Web 管理界面）时，**卡到完全无法使用**。

但同一台设备，用局域网 IP 直接访问，**秒开、毫无延迟**。

| 访问方式 | 地址 | 体验 |
|----------|------|------|
| EasyTier 虚拟 IP（走 SOCKS5） | `http://10.126.126.3:6185` | ❌ 卡到无法使用 |
| 局域网 IP（直连） | `http://192.168.1.238:6185` | ✅ 秒开 |

> 两个地址指向**同一台设备**的**同一个服务**，只是访问路径不同。

---

## 排查过程

### 第一步：确认 EasyTier 网络本身是否正常

在 VPS 上执行 `easytier-cli peer`，查看 VPS 到目标设备的连接状态：

```
| ipv4            | hostname  | cost | lat(ms) | loss | tunnel |
|-----------------|-----------|------|---------|------|--------|
| 10.126.126.3/24 | MAC-MINI  | p2p  | 54.93   | 1.0% | udp    |
```

**结论**：
- ✅ VPS 与 MAC-MINI 之间是 **P2P 直连**（不是中继），路径最优
- ✅ 延迟 **55ms**，正常范围
- ⚠️ 丢包率 **1%**，偏高但不是主要问题

### 第二步：从 VPS 测试单次 HTTP 请求

在 VPS 上直接通过本地 SOCKS5 访问目标服务：

```bash
curl -x socks5://127.0.0.1:15555 http://10.126.126.3:6185 \
  -o /dev/null -w "总耗时: %{time_total}s | 首字节: %{time_starttransfer}s\n"
```

```
总耗时: 0.299s | 首字节: 0.298s
```

**结论**：
- ✅ 单次请求 **300ms**，完全可接受
- ⚠️ 但这只是**一个请求**，浏览器打开一个网页远不止一个请求

### 第三步：问题定位 — 为什么浏览器会卡？

单次请求 300ms 没问题，那为什么浏览器打开网页会卡到无法使用？

**因为一个网页不是一个请求。**

---

## 根本原因：SOCKS5 的逐请求握手开销

### 一个网页的真实加载过程

当你在浏览器打开 `http://10.126.126.3:6185` 时，实际发生的事情：

```
浏览器请求网页 HTML      →  1 个请求
HTML 引用的 CSS 文件      →  2-5 个请求
HTML 引用的 JS 文件       →  3-10 个请求
JS 加载后发起的 API 调用  →  5-15 个请求
图片、字体、图标          →  5-20 个请求
────────────────────────────────────────
总计                      →  15-50 个请求
```

一个普通的 Web 管理界面，浏览器需要发出 **20-50 个 HTTP 请求**才能完整渲染。

### 每个请求的完整链路

关键在于：每一个请求都要走完整的代理链路，包括多次网络握手。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    一个 HTTP 请求的完整旅程                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  📱 手机                                                            │
│   │                                                                 │
│   │ ① TCP 握手：手机 ↔ VPS                    ~50ms × 2 = 100ms    │
│   │   （SYN → SYN/ACK → ACK）                                      │
│   │                                                                 │
│   │ ② SOCKS5 握手：协商认证 + 连接请求         ~50ms × 2 = 100ms    │
│   │   （手机告诉 VPS：请帮我连 10.126.126.3:6185）                  │
│   │                                                                 │
│   │ ③ TCP 握手：VPS ↔ 目标设备（穿 EasyTier）  ~55ms × 2 = 110ms   │
│   │   （VPS 通过 EasyTier 虚拟网络连接目标设备）                     │
│   │                                                                 │
│   │ ④ HTTP 请求/响应                            ~105ms × 2 = 210ms  │
│   │   （请求从手机出发 → VPS → 目标，响应原路返回）                  │
│   │                                                                 │
│   ▼                                                                 │
│  总计：每个请求 ≈ 520ms                                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> 其中手机 → VPS 的延迟按 50ms 估算，实际取决于你的网络环境。

### 乘以请求数量

浏览器通常对同一域名最多保持 **6 个并发连接**。假设页面需要 30 个请求：

```
30 个请求 ÷ 6 并发 = 5 轮请求
每轮 ≈ 520ms
5 轮 × 520ms = 2.6 秒
```

这还是理想情况。实际上：

- **1% 丢包**会触发 TCP 重传，重传超时通常 **200ms-1s**
- 部分请求有**依赖关系**（必须等上一个完成才能发下一个）
- SOCKS5 连接**不能复用**，每个请求都要重新握手

最终体验：页面加载 **5-10 秒以上**，交互操作每次点击都要等，**卡到无法使用**。

### 对比：直接访问为什么秒开？

当你用局域网 IP `http://192.168.1.238:6185` 访问时：

```
┌──────────────────────────────────────────────┐
│           局域网直连的请求旅程                 │
├──────────────────────────────────────────────┤
│                                              │
│  📱 手机                                      │
│   │                                          │
│   │ ① TCP 握手：手机 ↔ 设备     ~1ms × 2     │
│   │ ② HTTP 请求/响应             ~1ms × 2     │
│   │                                          │
│   ▼                                          │
│  总计：每个请求 ≈ 4ms                         │
│                                              │
│  30 个请求 ÷ 6 并发 = 5 轮                    │
│  5 轮 × 4ms = 20ms → 秒开                    │
│                                              │
└──────────────────────────────────────────────┘
```

**520ms vs 4ms**，差距 **130 倍**。这就是为什么一个秒开、一个卡死。

## 解决方案：nginx 反向代理

### 核心思路

VPS 本身就在 EasyTier 虚拟网络里（它运行着 EasyTier 节点），可以**直接**访问 `10.126.126.3`，根本不需要经过 SOCKS5。

我们在 VPS 上加一层 nginx 反向代理：

- 手机直接用 HTTP 连到 VPS
- nginx 转发请求到内网设备
- nginx 和内网设备之间保持**长连接复用**

### 改进后的请求链路

```
┌──────────────────────────────────────────────────────────────────────┐
│                    改进后：一个 HTTP 请求的旅程                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  📱 手机                                                             │
│   │                                                                  │
│   │ ① TCP 握手：手机 ↔ VPS nginx               ~50ms × 2 = 100ms   │
│   │   （普通 HTTP 连接，后续请求复用此连接）                          │
│   │                                                                  │
│   │ ② HTTP 请求 → nginx → 内网设备              ~55ms × 2 = 110ms   │
│   │   （nginx 通过 EasyTier TUN 接口直连设备）                       │
│   │   （连接保持复用，不需要每次重新握手）                            │
│   │                                                                  │
│   ▼                                                                  │
│  首次请求 ≈ 210ms                                                    │
│  后续请求 ≈ 110ms （TCP 连接已建立，跳过握手）                       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 为什么 nginx 方案快得多？

| 对比项 | SOCKS5 方案 | nginx 方案 | 差距 |
|--------|-------------|------------|------|
| 手机 → VPS 握手 | 每个请求都要 TCP + SOCKS5 握手 | 一次 TCP 握手，后续 HTTP Keep-Alive 复用 | 省 ~200ms/请求 |
| VPS → 内网设备握手 | 每个请求都要新建 TCP 连接 | nginx upstream keepalive，连接保持复用 | 省 ~110ms/请求 |
| 每请求耗时 | ~520ms | 首次 ~210ms，后续 ~110ms | **快 3-5 倍** |
| 30 个请求总耗时 | ~2.6s+ | ~0.7s | **快 4 倍** |
| 考虑丢包重传 | 更多连接 = 更多丢包影响 | 更少连接 = 更少丢包影响 | 稳定性更好 |

### 关键优化点

**1. HTTP Keep-Alive（手机 → VPS）**

浏览器和 nginx 之间的 TCP 连接只建立一次，后续所有请求复用同一条连接。省去了每个请求的 TCP 握手和 SOCKS5 协商。

**2. Upstream Keepalive（VPS → 内网设备）**

nginx 到内网设备（通过 EasyTier TUN 接口）的连接也保持存活。`keepalive 16` 表示维护 16 条持久连接。一个页面加载的 30 个请求，可能只需要建立 1-2 次 TCP 连接。

**3. 无 SOCKS5 开销**

VPS 直接通过 EasyTier TUN 网络接口访问 `10.126.126.3`，走的是操作系统的网络栈，不经过 SOCKS5 代理。每个请求省去了 SOCKS5 协商的 2 次往返。

**4. WebSocket 支持**

nginx 配置了 `Upgrade` 和 `Connection` 头的自动处理。如果 Web 管理界面使用 WebSocket（很多现代 Web UI 都用），连接会自动升级为 WebSocket 长连接，无需反复建立。

### 改进前：SOCKS5 方案

```
📱 手机浏览器
 │
 │  访问 http://10.126.126.3:6185
 │  Clash 匹配规则 → 走 SOCKS5 代理
 │
 ▼
☁️ VPS (SOCKS5:15555)
 │
 │  每个 HTTP 请求：
 │  1. 接收 SOCKS5 握手
 │  2. 新建 TCP 连接到目标
 │  3. 转发数据
 │  4. 请求结束，关闭连接  ← 无法复用！
 │
 ▼
🖥️ MAC-MINI (10.126.126.3:6185)
```

### 改进后：nginx 反向代理

```
📱 手机浏览器
 │
 │  直接访问 http://VPS公网IP:6185
 │  普通 HTTP 请求，无需 Clash 介入
 │  TCP 连接复用（Keep-Alive）
 │
 ▼
☁️ VPS (nginx:6185)
 │
 │  nginx 处理：
 │  1. 从连接池取一条到目标的连接（或新建）
 │  2. 转发请求
 │  3. 返回响应
 │  4. 连接放回池中  ← 下次直接复用！
 │
 ▼
🖥️ MAC-MINI (10.126.126.3:6185)
    通过 EasyTier TUN 接口直连
```

## 使用方式

### 部署

nginx 已集成在 `docker-compose.yml` 中，执行以下命令即可启动：

```bash
cd docker
docker compose up -d
```

### 访问

直接在浏览器中打开 **VPS 公网 IP + 端口**：

| 原地址（SOCKS5，慢） | 新地址（nginx，快） |
|----------------------|---------------------|
| `http://10.126.126.3:6185` | `http://VPS公网IP:6185` |
| `http://10.126.126.3:23000` | `http://VPS公网IP:23000` |

**不需要走 Clash**，浏览器直接打开即可。

### 添加更多端口

编辑 `docker/nginx.conf`，按照已有格式添加 upstream 和 server 块：

```nginx
# 1. 添加 upstream
upstream my_device_PORT {
    server 10.126.126.X:PORT;    # ← 替换为目标设备虚拟 IP 和端口
    keepalive 16;
}

# 2. 添加 server
server {
    listen PORT;                  # ← VPS 上监听的端口（通常与目标端口相同）
    location / {
        proxy_pass http://my_device_PORT;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_connect_timeout 10s;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
    }
}
```

然后重启 nginx 容器：

```bash
docker compose restart easytier-nginx
```

### 安全提醒

nginx 代理的端口会暴露在公网上。建议配合防火墙限制来源 IP：

```bash
# 只允许你的 IP 访问
sudo iptables -A INPUT -p tcp --dport 6185 -s YOUR_IP -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 6185 -j DROP
```

## 适用场景

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 访问内网 HTTP/Web 服务 | ✅ nginx 反向代理 | 连接复用，体验流畅 |
| SSH 连接内网设备 | SOCKS5 即可 | 单连接长会话，握手开销可忽略 |
| 访问临时/不固定的端口 | SOCKS5 | nginx 需要预配置每个端口 |
| 在同一局域网内 | 直接用局域网 IP | 不需要任何代理 |

## 总结

- **卡顿原因**：SOCKS5 对每个 HTTP 请求都要完整握手，Web 页面几十个请求的延迟叠加导致卡死
- **解决方案**：在 VPS 上加 nginx 反向代理，利用连接复用消除逐请求握手开销
- **效果**：页面加载速度提升 3-5 倍，从"卡到无法使用"变为"正常可用"
- **使用方式**：浏览器直接访问 `http://VPS公网IP:端口`，无需 Clash 介入
